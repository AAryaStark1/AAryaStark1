本篇笔记是在阅读《Redis设计与实现》这本书的阅读笔记，旨在对本书的要点进行一个简易的总结。主要包括以下三个部分：

一、数据结构与对象
redis本身是一个基于内存的键值对数据库，相比于磁盘在io读写上的速度有很大的提升，同时，redis也设置了独特的数据结构，用于加快redis对节点的访问速度，以下是这本书中一些主要的数据结构类型。

1、sds字符串
sds是简单动态字符串的意思，是redis中应用最为广泛的数据类型。一个简单动态字符串的结构定义包括一个len的长度信息，free用于表示sds中为使用的字节的数量，同时还包括一个字节数组，用于保存字符串信息。sds保留了c字符串的编码习惯，使用了\0作为字符串的结尾，兼容了很多c字符串的api。这样设计的好处有：
·常数时间内获得字符串的长度。
·杜绝缓冲区的溢出，在进行字符串拼接等操作的时候，会触发sds的扩容策略，实现动态的扩容，防止缓冲区的溢出。
·减少修改字符串带来的内存分配的次数，通过free保罗的空闲内存区域，可以减少内存分配的次数。
·二进制安全，redis中的sds还能够存储\0的空字符，通过其他的编码方式进行解析。

2、redis中的链表结构
redis的链表结构是通过双向链表实现的，list结构保留head和tail头尾指针，还有一个长度信息表示链表中节点的个数，每个节点使用一个listnode进行存储，记录着节点的value值。

3、redis中的字典
redis中的字典用于存储关联的数据，key，value的键值对。字典的实现主要通过哈希表来实现，一个字典中的实现包括以下的数据结构，两个哈希表dictht[2]，trehashidx这两个哈希表用于保留具体的键值对信息，同时另一个哈希表用于在扩容时使用，trehashidx索引用于跟踪rehash的进度。哈希表中的数据主要包括：双重指针，指向具体哈希槽数组的指针，哈希表的大小，用于之时哈希槽的大小，掩码，通常是哈希表的大小-1，一般用于位于操作计算索引值，哈希表节点的数量，用于指示rehash的时机。每一个哈希表节点都包括一个key值和具体的value值，以及指向下一个哈希白哦节点的指针，用于在哈希槽中形成链表。

值得注意的是哈希表解决键冲突的问题，使用链地址法来解决hash冲突，这里有个技巧是将新的节点放置在哈希槽的队头中，在实现速度和时间局部性上都有好的效果。随着哈希表中键值对的个数越来越多，负载因子会越来越高，扩容的策略是选择第一个大于两倍哈希表使用量的2的幂次数，同时交换两个哈希表实现具体的扩容操作，redis中的扩容是一种渐进式的扩容策略，在对扩容后的哈希表进行查询操作的时候，redis会先从原先的哈希表进行查找，并将该哈希槽的的所有元素移动到新的哈希表中，并进行rehashidx的计数，直到将所有的哈希表元素移动到新的哈希表中，才会将这个索引的值置为-1，表示扩容成功。

4、跳跃表的实现
跳跃表是redis中zset的实现方式，支持在logN的时间复杂度中，最坏是0N的时间复杂度上完成查找操作，跳跃表通过两个数据结构进行实现，一个是zskiplist结构和一个zskiplistnode结构。
zskiplist结构中包括一个header头部信息，指向跳跃表中的表头节点，tail表示跳跃表中的队尾节点，length表示跳跃表中的节点数量，level表示跳跃表中最高的层数。更为核心的数据结构是一个zskiplistnode结构，其中记录着一个后向的指针，一个前向level数组，其中每一个元素包括一个前向的zskiplistnode的指针和一个节点的跨度，还包括具体存储的成员对象以及一个score节点的分数。在跳跃表中，每个节点的分数可以是相同的，但是数据对象不能是相同的。通过跳跃表节点的level层数通过算法来实现，确保都是0-32之间的整数，在进行遍历的时候，可以通过level数据进行跨节点的访问，同时通过level的跨度实现查找对象的排位记录。

二、单机reids的机制
redis是一个键值对的数据库，通过redisServer进行定义，其中保留了redisdb数据，每一个数组元素表示一个redis的数据库，可以通过select *进行选择数据库。在数据库中，每一个的redisDb都有有一个dict的字典进行保留的，其中记录着所有的键值对信息，所有对redis中数据的增删改查本质上都是对这个dict字典的修改或是查询。同时redisDb中同样保留一个键空间expire，其中记录着对应键的过期时间，其中过期时间由long long的类型进行定义，在设置redis键的时候可以使用一个expire的指令向这个键空间中添加元素。

针对过期键的删除策略，常见的删除策略主要包括以下几种，定时删除，定期删除和惰性删除。实际上，redis是通过惰性删除和定期删除两个方式来实现的，惰性删除通过在进行redis数据操作的时候，会先调用expireIFNeeded来检查过期键是否过期...定期删除的策略是使用redis每一段时间中从数据库中获取一部分的键盘，删除其中过期的键，直到过期的键占取样的样本数量的阈值之下，则定期删除实现完毕。AOF和RDB持久化的实现中，当使用RDB对redis进行持久化操作之后，会检查键的过期时间，如果过期了，则不加入到RDB的快照中。在进行aof重写的时候，面对过期的键不会将过期的键进行删除，在进行惰性删除或是定期删除的时候，会将删除的语句加入的aof文件中，进行显示的同步。同时，在进行aof重写缓冲区的时候，则会进行过期键的检查，过期键不会被加入的aof持久化文件中。

RDB持久化
RDB持久化是Redis中默认的持久化方式，其核心思路是定期备份，创建的RDB是二进制文件，通过压缩算法进行压缩存储，在redis重启的时候可以通过redis快照RDB进行恢复。RBD持久化通过命令save和bgsave来实现，其中save命令是一个阻塞的命令，会阻塞当前任务的执行。bgsave是非阻塞的命令，通过后台创建一个子进程进行rdb的写入，实现的原理是典型的写时复制的技术。RDB一般通过显示调用或者是定时器进行执行。
AOF持久化
AOF是将客户端的每一个写操作命令都记录到日志中，追加到后缀为aof的文件末尾，在Redis服务器重启时，会加载并运行aof文件的所有命令，以达到恢复数据的目的。redis开启aof缓冲的时候会开启一个缓冲区，redis进行时间循环接收客户端上的命令的时候，会将命令追加到aof的缓冲区末尾。根据设置的同步策略将缓冲区的文件写入到硬盘中，文件同步的策略可以分为以下几种，always,everysecond,no。同时由于AOF是追加的策略，在redis运行过久后会导致aof文件体积过大，为了缓解这一问题，reids实现了aof重写策略，aof重写策略名称上会比较有歧义，实际上于原有的aof文件关系不大，而是会扫描整个redis数据库，将文件进行set写入，这一步骤也会开启一个后台的进程进行aof文件的重写，使得在执行aof重写期间也能够接收到新的客户端的命令，实现的原理也是写时复制的原理。通过将写入的命令记录到aof重写缓冲区中，在完成aof重写后将aof缓冲区中的数据附加到磁盘的文件中，实现redis数据库的同步。
三、集群redis的实现
1、redids的主从复制
Redis的主从复制是一种高可用性解决方案，它允许数据从主节点（Master）复制到一个或多个从节点（Slave）。从服务器通过执行slaveof命令执行主从复制，将一个redis实例作为另一个redis的从服务器。在执行salveof指令时，从redie节点会执行sync同步命令，以下是一些具体的实现：
从服务器向主服务器发送sync命令，主服务器接收执行bgsave指令，在fork出来的子进程中执行生成rdb文件，同时将在生成的过程中将主redis节点中接收到的指令写入到缓冲区中，随后主redis节点将rdb文件发送到从redis节点中从服务器进行加载，随后主服务器将缓冲区中的数据发送的从服务器中，进行数据的同步。后续主从服务器之间会进行命令同步，将指令发送的从服务器上实现主从服务器上数据的一致。

上述机制实现了主从服务器之间的同步，但是在网络断开的场景下，从服务器会重新执行sync同步命令，但是sync同步命令是一个消耗比较大的命令，在网络状况不好的情况下的表现不好。后续通过了psync命令改善了这一情况，实现了部分从同步。上述的实现主要依三个机制，主从服务器的复制偏移量，主服务器的复制积压缓冲区，主服务器的运行id。

主从偏移量用于表示主服务器和从服务器的同步进度，相同的偏移量总是表示相同的同步情况，这里的偏移量会持久保存。复制积压缓冲区是一个固定长度的先进先出的队列，当主服务器进行命令传播的时候，不仅会将写命令发送到所有的从服务器，还会将写写命令入队到复制积压缓冲区中，当从服务器进行psync进行同步时会带有offset偏移信息，会进行判断该命令是否在复制积压缓冲区中，如果存在则进行部分数据的同步，将该偏移量之后的数据发送的客户端中，执行相应的命令。服务器id用于表示当前的客户端是否连接了主服务器，判断是否进行全量同步或者是增量同步。
2、redis的哨兵模式
哨兵模式时redis高可用的一个解决方案，通过一个或多个Sentinel实例进行监控一个或多个redis实例，以及所属这个主服务器中所有的从服务器。Sentinel是运行在特殊模式下的redis实例，使用了一套特殊的代码。
具体的实现如下：
哨兵节点通过通过读取配置文件以及向主服务器发送INFO命令获得主服务器和从服务器的地址信息，并创建命令连接和订阅连接。在一般情况下，Sentinel以每10秒一次的频率进行对主服务节点和从服务节点的监控，当主服务节点断开时，发送info命令的频率会增长到一秒一次，哨兵节会调用ping命令向主服务器，从服务器和其他哨兵节点发送ping命令，当一个实例在指定的时长里，都没有向哨兵节点发送有效的回复的时候，会标记这个主节点为主观下线，随后会向其他的哨兵节点进行询问，看它们是否也同意该节点主观下线，当有足够多的哨兵节点同意主观下线的时候，该节点会被判断为客观下线，随后进行故障转移。

当主服务器被标记为客观下线后，哨兵之间会进行一次选举，选择一个哨兵来执行故障转移操作，被选举出的领导者哨兵会执行故障转移操作，将从服务器提升为主服务器。这里会使用一个raft共识算法，确保只有一个哨兵节点会称为哨兵leader。判断为主服务节点主观下线的哨兵节点会像其他的哨兵节点发起选举，要求推选该哨兵节点为哨兵leader，同时在一个epoch里，哨兵节点只能投出一票，确保在主服务器发生故障时，快速且一致地选举出一个新的领导者来负责故障转移。

哨兵leader会对从服务器节点中选择一个提升为主服务器，实现的优先级顺序为：网络优先级顺序，确保从服务器都是最新和主服务器进行通信过的；偏移量最大，运行id最小的一个从服务器。选择成功后哨兵节点会发送slave no one命令升级为主服务器，将剩余的从服务器中变为新主服务器的从服务器，如果原先的主服务重新连接之后，会将该主服务器作为新的从服务器进行使用。
3、redis的集群模式
Redis集群模式是一种分布式数据库解决方案，它通过将数据自动分割到多个节点上，提高了扩展性和可用性。
Redis集群通过数据分片来实现数据的分布式存储。数据被分割成多个片段，称为槽（slots），每个槽包含一部分数据。Redis集群中有16384个槽，每个节点负责其中一部分槽位。客户端需要知道数据被存储在哪个节点上。Redis集群通过CRC16算法对key进行计算，将计算结果对16384取余，这样每个key都会对应一个哈希槽。每个集群节点中，都知道所有的redis集群的槽指派的信息，通过一个数组来实现，实际上使用char数组，长度为16384/8，设置特定的比特位，表明该redis实例支持的槽位。同时每个集群节点都需要一个共享的数据结构用来表示槽分配信息，ClusterState其中有一个slots数组，每一个元素是一个ClusterNode的指针，如果是null表示是一个为指派的槽，ClusterNode中保留有节点的ip和端口，如果执行的指令在错误的槽中，会返回一个moved的错误码，通过再次定位将指令发送到正确的redis节点，实现结果的返回。而redis集群还可以通过热加载的方式，将新的redis节点加入到集群中，主要通过修改上述的ClusterState的槽指派信息修改具体的对应的槽的reids节点。
