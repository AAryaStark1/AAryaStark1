基于io_uring和C++20协程实现的现代高性能web服务器

本片笔记是对C++传统项目webserver的改进，笔者在面试的时候经常被问起对webserver有何改进，往往说不上来，或是没有亮眼的表现，特此找到一篇对WebSerVer的重大改进项目，使用io_uring异步机制实现了真正的proactor模型，使用了C++20的协程支持，借鉴了golang中协程调度的思路，通过单线程实现多协程的高效调度，通过无锁队列实现日志的高效写入，并改进了小根堆计数器的思路，实现了事件轮算法进行客户端的超时管理，避免了小根堆logn的时间复杂度瓶颈，原作者链接https://github.com/AomaYple/webServer

一、C++20协程调度
这里是对C++20协程的应用，构建了基础的协程调度框架，配合IO_URING的异步实现。以下是对协程库的实现：

1、Promise实现

class promise_type {
public:
    [[nodiscard]] auto get_return_object() -> Task;

    [[nodiscard]] constexpr auto initial_suspend() const noexcept { return std::suspend_always{}; }

    [[nodiscard]] constexpr auto final_suspend() const noexcept { return std::suspend_always{}; }

    auto unhandled_exception() const -> void;

    auto setSubmission(const Submission &submission) noexcept -> void;

    [[nodiscard]] auto getSubmission() const noexcept -> const Submission &;

    auto setOutcome(Outcome outcome) noexcept -> void;

    [[nodiscard]] auto getOutcome() const noexcept -> Outcome;

private:
    Submission submission;
    Outcome outcome;
};

在这里promise_type类为Task实现的内部类，为所有的Task对象所共用，pormise_type中实现为两个自定义的数据结构，分别对应提交给IO_URING提交队列和完成队列的数据，分别由不同的异步调用(如网络IO)所设置，并在异步调用结束后根据IO_URING完成队列中的返回值进行设置。其内部实现为：

struct Submission {
    enum class Type : unsigned char { write, accept, read, receive, send, cancel, close };
    struct Write {
        std::span<const std::byte> buffer;
        unsigned long offset;
    };
    struct Accept {
        sockaddr *address;
        socklen_t *addressLength;
        int flags;
    };
    struct Read {
        std::span<std::byte> buffer;
        unsigned long offset;
    };
    struct Receive {
        std::span<std::byte> buffer;
        int flags;
        int ringBufferId;
    };
    struct Send {
        std::span<const std::byte> buffer;
        int flags;
        unsigned int zeroCopyFlags;
    };
    struct Cancel {
        int flags;
    };
    struct Close {};
    int fileDescriptor;
    unsigned int flags;
    unsigned short ioPriority;
    unsigned long userData;
    std::variant<Write, Accept, Read, Receive, Send, Cancel, Close> parameter;
};

上述的是数据类型中fileDescriptor，flags，ioPriority，userData都是IO_URING提交队列中的必要项，parameter为不同异步IO所需的特有参数，对于write、send则需要传入用户态的缓冲区，文件偏移量；对于Accept则需要传入地址信息，以及socket对于的关键字，Receive需要传入用户态接收的缓冲区(这里实际不需要)，标志位，缓存区id(用于实际缓冲区查找);对于send函数需要额外传递一个用于表示是否进行零拷贝的偏执为，close和cancel则按照正常的参数传递。

struct Outcome {
    int result;
    unsigned int flags;
};

Outcome中存储中完成队列中返回的数据，只有一个result表示send，write，send或recv中发出或接收的字节，flags为返回的IO_URING标志位。

2、Task实现

class Task {
public:
    explicit Task(std::coroutine_handle<promise_type> handle) noexcept;
    Task(const Task &) = delete;
    Task(Task &&) noexcept;
    auto operator=(const Task &) -> Task & = delete;
    auto operator=(Task &&) noexcept -> Task &;
    ~Task();
    [[nodiscard]] auto getSubmission() const -> const Submission &;
    auto resume(Outcome outcome) const -> void;
private:
    auto destroy() const -> void;
    std::coroutine_handle<promise_type> handle;
};
Task::~Task() { this->destroy(); }
auto Task::getSubmission() const -> const Submission & { return this->handle.promise().getSubmission(); }
auto Task::resume(const Outcome outcome) const -> void {
    this->handle.promise().setOutcome(outcome);
    this->handle.resume();
}
auto Task::destroy() const -> void {
    if (this->handle) this->handle.destroy();
}

这里的实现亮点是在析构函数中调用了destroy()函数，实现了协程的销毁，也是RAII的一种实现，但是注意不要手动的调用destory函数，避免多次析构造成为程序的段错误，支持这里RAII实现的一个支持是使用了

constexpr auto Task::promis_type::final_suspend() const noexcept { return std::suspend_always{}; }

协程的最终挂起，使得系统实现有充足的时间进行IO_URING的结果返回和状态保留。

在调用

auto resume(Outcome outcome) const -> void

函数的时候，首先通过promise对象中的setOutcome接口实现结果的保留，随后进行协程的恢复执行

3、Awater实现

class Awaiter {
public:
    explicit Awaiter(const Submission &submission) noexcept;
    [[nodiscard]] constexpr auto await_ready() const noexcept { return false; }
    auto await_suspend(std::coroutine_handle<Task::promise_type> handle) -> void;
    [[nodiscard]] auto await_resume() const -> Outcome;
private:
    std::coroutine_handle<Task::promise_type> handle;
    Submission submission;
};
Awaiter::Awaiter(const Submission &submission) noexcept : submission{submission} {}
auto Awaiter::await_suspend(const std::coroutine_handle<Task::promise_type> handle) -> void {
    this->handle = handle;
    this->submission.userData = std::hash<std::coroutine_handle<Task::promise_type>>{}(this->handle);
    this->handle.promise().setSubmission(this->submission);
}
auto Awaiter::await_resume() const -> Outcome { return this->handle.promise().getOutcome(); }

Awaiter对象中保留一个协程句柄，和一个提交项目，提交项由需要提交的异步调用所设置，handle用于保留当前的协程，在需要挂起的时候通过调用await_suspend将当前的协程句柄进行保留，同时设置对协程句柄进行hash处理，生成唯一的id，并通过引用传入的promise对象中。await_resume为对象恢复时需要执行的函数，将结果值返回协程中，这里是将promise中的Outcome进行返回，具体执行的时间在调用对应的task的resume函数时，先设置好OutCome结果值，后恢复协程的执行，返回co_wait的返回值。

二、IO_URINGd对异步的执行
在需要执行异步IO的时候，同构co_wait函数进行协程挂起，根据具体的系统调用设置不同的Awaiter对象，如客户端的receive和send函数：

auto Client::receive(const int ringBufferId) const noexcept -> Awaiter {
    return Awaiter{
        Submission{
                   this->getFileDescriptor(),
                   IOSQE_FIXED_FILE | IOSQE_BUFFER_SELECT,
                   0, 0,
                   Submission::Receive{std::span<std::byte>{}, 0, ringBufferId},
                   }
    };
}

auto Client::send(const std::span<const std::byte> data) const noexcept -> Awaiter {
    return Awaiter{
        Submission{
                   this->getFileDescriptor(),
                   IOSQE_FIXED_FILE, 0,
                   0, Submission::Send{data, 0, 0},
                   }
    };
}

顺序构建出需要提交的Submission，回到主函数的执行流程中，主函数中都通过一个submit函数进行协程的创建，

auto Scheduler::submit(std::shared_ptr<Task> &&task) -> void {
    task->resume(Outcome{});
    this->ring->submit(task->getSubmission());
    this->tasks.emplace(task->getSubmission().userData, std::move(task));
}

task->resume(Outcome{})用于初始化挂起的协程的执行，流程回到协程的函数体，在执行cowait设置协程中promise的提交项后，进行协程的挂起，回到主函数的this->ring->submit(task->getSubmission())将Submission对象进行解析，提交到IOURING的提交队列中，这里的执行逻辑时异步提交的关键。

auto Ring::submit(const Submission &submission) -> void {
    io_uring_sqe *const sqe{this->getSqe()};

    switch (static_cast<Submission::Type>(submission.parameter.index())) {
        case Submission::Type::write:
            {
                const auto [buffer, offset]{std::get<Submission::Write>(submission.parameter)};
                io_uring_prep_write(sqe, submission.fileDescriptor, buffer.data(), buffer.size(), offset);

                break;
            }
        ...
            break;
    }
    io_uring_sqe_set_flags(sqe, submission.flags);
    sqe->ioprio |= submission.ioPriority;
    io_uring_sqe_set_data64(sqe, submission.userData);
}

通过上liburing中提供的函数接口，实现sqe提交项的设置，并设置好sqe的标志位置，优先级和64位的自定义数据。

提交sqe请求之后，通过调用iouring的wait函数，当有一定数量的IO请求完成时后调用frame()决定对应协程的恢复执行，这里的关键是使用了userdata唯一标识了一个task协程对象，同时在调度器中使用了

std::unordered_map<unsigned long, std::shared_ptr<Task>> tasks；

进行了task对象的管理。调用task的resume对象实现了协程的结果的设置，并恢复对应协程的执行，调用awaiter的resume函数返回co_wait的结果，供给后续的协程函数体执行。

auto Scheduler::frame() -> void {
    const int completionCount{
        this->ring->poll(
            [this](const Completion &completion) {
                if (completion.outcome.result != 0 || (completion.outcome.flags & IORING_CQE_F_NOTIF) == 0) {
                    this->currentUserData = completion.userData;
                    const std::shared_ptr task{this->tasks.at(this->currentUserData)};
                    task->resume(completion.outcome);
                }
            }
        )
    };

    this->ring->advance(this->ringBuffer.getHandle(), completionCount, this->ringBuffer.getAddedBufferCount());
}

三、对无锁日志类的实现
见日志类并发实现中，完成了多消费者单生成者的无锁队列实现，在对模拟日志的实验中于boost实现的无锁队列效率相当，明显过于基于锁的阻塞队列的日志实现。日志类维持一个日志队列和一个用户态缓冲区

std::lock_free_queue<Log> logs;
std::vector<std::byte> buffer;

在执行pop函数时，会一值移动队头指针，单指针指向的元素为空时候，代表队列中的元素为空，pop失败，队列所有的日类元素以及拷贝到buff中了，后续调用write系统调用进行异步的写入。
