本篇笔记主要对双重检查锁实现单例模式是否是线程安全的探讨,总结对单例模型设计学习的一系列过程,以及一些从CppCon中的学习学习感悟。

1、饿汉式的单例实现

template <typename T>
class Singleton {
public:
    static T& getInstance() {
        return instance;
    }

private:
    Singleton() {}
    ~Singleton() {}

    Singleton(const Singleton&) = delete;
    Singleton& operator=(const Singleton&) = delete;

    static T instance;  // 静态实例
};

template <typename T>
T Singleton<T>::instance;

饿汉式单例模式在类加载时就初始化实例，因此是线程安全的，不需要额外的同步机制。

二、懒汉式单例模式
懒汉式单例模式在第一次使用时才初始化实例，这样可以避免不必要的初始化开销。但在线程安全性上有了更高的要求。
非线程安全的懒汉式单例模式

class Singleton {
public:
    static Singleton* getInstance() {
        if (instance == nullptr) {
            instance = new Singleton();
        }
        return instance;
    }

private:
    Singleton() {}
    ~Singleton() {}

    Singleton(const Singleton&) = delete;
    Singleton& operator=(const Singleton&) = delete;

    static Singleton* instance;
};
Singleton* Singleton::instance = nullptr;

针对instance的判空在多线程并发情况会发生创建多个Singleton对象的问题,导致内存泄漏。

加锁实现的线程安全的单例模式

template <typename T>
class Singleton {
public:
    static T* getInstance() {
        std::lock_guard<std::mutex> lock(mutex);
        if (instance == nullptr) {
            instance = new T();
        }
        return instance;
    }
private:
    Singleton() {}
    ~Singleton() {}
    Singleton(const Singleton&) = delete;
    Singleton& operator=(const Singleton&) = delete;

    static T* instance;
    static std::mutex mutex;
};
template <typename T>
T* Singleton<T>::instance = nullptr;

template <typename T>
std::mutex Singleton<T>::mutex;

通过lock(mutex)加锁判断只有一个线程能进入临界区,并通过lock_guard加锁解锁保证所有线程的可见性,这里式线程安全的实现。但是每次在调用getInstance()函数的时候都会进行加锁判断,造成很大程度的浪费。

双重检查锁实现的懒汉模式

template <typename T>
class Singleton {
public:
    static T* getInstance() {
        if (instance == nullptr) {
            std::lock_guard<std::mutex> lock(mutex);
            if (instance == nullptr) {
                instance = new T();
            }
        }
        return instance;
    }
private:
    Singleton() {}
    ~Singleton() {}
    Singleton(const Singleton&) = delete;
    Singleton& operator=(const Singleton&) = delete;

    static T* instance;
    static std::mutex mutex;
};
template <typename T>
T* Singleton<T>::instance = nullptr;

template <typename T>
std::mutex Singleton<T>::mutex;

典型的java的实现模式,但是java可以通过volatile来保证内存的可见性,c++的双重检查模式也可能存在一定的线程不安全性.本质上instance = new T()申请内存进行初始化的过程并不是一个原子性的操作。但是在多线程并发的情况下,对instance的判断返回依赖与上述创建的中间结果,其中就有可能导致另一个线程在未经加锁的情况下返回了未经初始化的单例,造成线程不安全的情况。本质上是因为多线程并行中一个线程的执行逻辑依赖于另一个线程的中间结果导致的线程不同步的情况。一个典型的instance = new T()操作可以分为以下三个步骤:

1:操作系统申请一块内存空间,返回一个指向给内存空间的指针
2:调用SingleTon的初始化函数进行内存的初始化
3:将上面的临时对象的指针拷贝到instance的用户对象中

上述发生的场景是操作系统的CPU执行优化,编译器的优化都有可能导致指令的重排,将1->2->3的执行顺序重排到1->3->2只有在当前创建的线程中真正访问的时候才会进行初始化,这里执行的时期是不可控的。在另一个线程访问instance对象时,并不能保证此时的单例对象的初始化工作已经完成了,另一个发生的原因是缓存架构的影响,在主存和CPU之间存在多级的缓存,缓存写入的时间也是不可控的,也可能推迟很多的时钟周期,总之instance指向的内存(主存)未完成初始化,造成未定义的行为。

C++11基于局部变量的懒汉模式

template <typename T>
class Singleton {
public:
    static T& getInstance() {
        static T instance;
        return instance;
    }
private:
    Singleton() {}
    ~Singleton() {}

    Singleton(const Singleton&) = delete;
    Singleton& operator=(const Singleton&) = delete;
};

C++11保证了对静态局部变量的初始化的线程安全性,因此实现线程线程安全的懒汉模式有了更加方便的实现方法。

三、C++11基于CRTP和可变参数模板实现的单例模式
上述创建的模式实现了线程安全性,但是其单例的创建,销毁都是不可控的,在多个单例模式相互依赖的情况下也无法进行良好的实现,以下是参考oneflow机器学习平台的一个参考实现方式。

template<typename T>
class SingleTon
{
private:
    SingleTon() = default;
    SingleTon(const SingleTon& other) = delete;
    SingleTon& operator=(const SingleTon& other) = delete;
    SingleTon(SingleTon&& other) = delete;
    SingleTon& operator=(SingleTon&& other) = delete;
    ~SingleTon(){
        SingleTonDestory();
    }
    static T** GetPPtr(){
        static T* singleton = nullptr;
        return &singleton;
    }
public:
    template<typename... Args>
    static void SingleTonCreate(Args... args){
        if(*GetPPtr() == nullptr){
            *GetPPtr() = new T(std::forward<Args>(args)...);
        }
        else{
            cout << "Has Created !" << endl;
        }
    }
    static T* SingleTonGet(){
        return *GetPPtr();
    }
    static void SingleTonDestory(){
        if(*GetPPtr()){
            delete *GetPPtr();
            *GetPPtr() = nullptr;
            cout << "Has Destory !" << endl;
        }
    }
};

class Example
{
    friend class SingleTon<Example>;
private:
    Example(int v, string n): val(v), name(n){};
    Example(const Example& other) = delete;
    Example& operator=(const Example& other) = delete;
    Example(Example&& other) = delete;
    Example& operator=(Example&& other) = delete;
private:
    int val;
    string name;
public:
    void Print(){
        cout << "Addr is " << this << endl;
        cout << "val is " << this->val << endl;
        cout << "name is " << this->name << endl;
    }
};

int main(){
    SingleTon<Example>::SingleTonCreate(10, "lee");
    Example* ptr = SingleTon<Example>::SingleTonGet();
    ptr->Print();
    SingleTon<Example>::SingleTonCreate(2, "tom");
    Example* ptr2 = SingleTon<Example>::SingleTonGet();
    ptr2->Print();

    SingleTon<Example>::SingleTonDestory();
}

上述通过提供的三个SingleTonCreate,SingleTonGet,SingleTonGet实现了单例模式的创建,获取和销毁,实现了单例的可控。并通过实现SingleTon的基类减少了单例类代码的重复。

关于CRTP奇异模板递归模式的应用,原理在模板基类中定义静态的单例函数,实现排除类的构建,通过friend友元实现两个类之间私有函数的可访问性。另一个CRTP的应用是通过enable_shared_from_this来传递自身的Shared_ptr。

一个典型的实现是：

class GoodCHUD : public std::enable_shared_from_this<GoodCHUD> {
public:
    void Print() {
        if (shared_from_this().use_count() == 3) {
            std::cout << "Doing stuff " << shared_from_this().use_count() << "\n";
        } else {
            std::cout << "Doing bad " << shared_from_this().use_count() << "\n";
        }
    }
    std::shared_ptr<GoodCHUD> GetSharedPtr() {
        return shared_from_this();
    }
    void StartAsyncOperation() {
        // 获取 shared_ptr，确保对象在异步操作期间不被销毁
        auto self = shared_from_this();
        std::thread t([self]() {
            std::this_thread::sleep_for(std::chrono::seconds(2));
            self->Print();
        });
        t.detach();
    }
    ~GoodCHUD() {
        std::cout << "GoodCHUD destroyed" << std::endl;
    }
};
int main() {
    GoodCHUD* chud = new GoodCHUD;
    std::shared_ptr<GoodCHUD> HUD1(chud);
    HUD1->StartAsyncOperation();
    auto HUD2 = HUD1->GetSharedPtr();
    std::this_thread::sleep_for(std::chrono::seconds(1));
    HUD1.reset();
    std::this_thread::sleep_for(std::chrono::seconds(3));
    return 0;
}

上述对shared_from_this的使用，主要体现在异步回调的场景，避免在多线程执行过程中shared_ptr引用计数为0导致内存释放，访问到意料之外的内存空间，引发未定义行为。本质原理是enable_shared_from_this模板类在内存维持一个weakptr的弱指针，当使用shared_ptr进行对象管理的时候，weak_ptr会与共享指针一样指向同一个引用计数块和对象指针。调用shared_ptr的时候会通过lock这个弱指针，返回一个shared_ptr，增加对象的引用计数，保证异步回调执行时期shared_ptr的线程安全。
