本篇笔记是对无锁数据结构进行内存释放的思考，基于链表的无锁队列需要依赖节点的next指针是否为空，或者是节点的计数是否为空来判断是否获得了无锁队列的队头或是队尾，由此带来的是严格的生命周期管理的挑战，否则一旦调用析构悬空的next指针就会造成未定义的行为。在linux系统5.10版本中实现的异步IOURING机制，其中就实现了基于环形数组的无锁队列，实现了高效的异步IO提交和收割。以下是对基于环形数组的无锁队列实现，实现的版本在https://github.com/DNedic/lockfree中。

一、基于循环数组的无锁数据结构

template <typename T, size_t size> class Queue {
    static_assert(std::is_trivial<T>::value, "The type T must be trivial");
    static_assert(size > 2, "Buffer size must be bigger than 2");

  public:
    Queue();

    bool Push(const T &element);

    bool Pop(T &element);

    std::optional<T> PopOptional();
  private:
    struct Slot {
        T val;
        std::atomic_size_t pop_count;
        std::atomic_size_t push_count;

        Slot() : pop_count(0U), push_count(0U) {}
    };
  private:
    Slot _data[size];
    std::atomic_size_t _r_count;
    std::atomic_size_t _w_count;
};

环形数组中的每一个元素为

struct Slot {
    T val;
    std::atomic_size_t pop_count;
    std::atomic_size_t push_count;
    Slot() : pop_count(0U), push_count(0U) {}
};

存放当前节点的实际数据，以及被pop和push的次数，初始化时都为0。

队列使用一个两个类型为atomic_size_t的原子变量_r_count、_w_count记录成功读取了写入的次数，用于标识push和pop实际操作的数组索引。队列主要提供了三个接口：

bool Push(const T &element);
bool Pop(T &element);
std::optional<T> PopOptional();

分别向队列中推入元素，弹出元素，以及存储弹出的元素。

二、push函数实现

template <typename T, size_t size>
bool Queue<T, size>::Push(const T &element) {
    size_t w_count = _w_count.load(std::memory_order_relaxed);

    while (true) {
        const size_t index = w_count % size;

        const size_t push_count = _data[index].push_count.load(std::memory_order_acquire);
        const size_t pop_count = _data[index].pop_count.load(std::memory_order_relaxed);

        if (push_count > pop_count) {
            return false;
        }

        const size_t revolution_count = w_count / size;
        const bool our_turn = revolution_count == push_count;

        if (our_turn) {
            if (_w_count.compare_exchange_weak(w_count, w_count + 1U, std::memory_order_relaxed)) {
                _data[index].val = element;
                _data[index].push_count.store(push_count + 1U, std::memory_order_release);
                return true;
            }
        } else {
            w_count = _w_count.load(std::memory_order_relaxed);
        }
    }
}

std::memory_order_acquire 是一种内存屏障（memory barrier），std::memory_order_acquire 确保了在这个读操作之后的所有内存操作都不会被重排到这个读操作之前。

在这个特定的场景中，std::memory_order_acquire 用于确保在读取 push_count 之前对共享数据的写操作（比如其他线程对 _data[index] 的写操作）都已经完成，并且这些写操作的结果对当前线程可见。这样可以保证当前线程在后续操作中使用 push_count 时，看到的是一个正确的、一致的状态。具体来说，假设 push_count 被用来判断某个元素是否已经被成功推入队列（例如，作为某种同步机制的一部分），那么使用 std::memory_order_acquire 确保了在读取 push_count 时，所有相关的写操作（例如，元素的推入操作）都已经完成并且可见。

这里使用_w_count.compare_exchange_weak(w_count, w_count + 1U, std::memory_order_relaxed)；在多线程并发的情况下只有一个线程能成功获得目标节点，并原子的增加_w_count和节点的push_count，用于节点的推进。

三、pop函数实现

template <typename T, size_t size> bool Queue<T, size>::Pop(T &element) {
    size_t r_count = _r_count.load(std::memory_order_relaxed);

    while (true) {
        const size_t index = r_count % size;

        const size_t pop_count = _data[index].pop_count.load(std::memory_order_acquire);
        const size_t push_count = _data[index].push_count.load(std::memory_order_relaxed);

        if (pop_count == push_count) {
            return false;
        }

        const size_t revolution_count = r_count / size;
        const bool our_turn = revolution_count == pop_count;

        if (our_turn) {
            if (_r_count.compare_exchange_weak(r_count, r_count + 1U, std::memory_order_relaxed)) {
                element = _data[index].val;
                _data[index].pop_count.store(pop_count + 1U, std::memory_order_release);
                return true;
            }
        } else {
            r_count = _r_count.load(std::memory_order_relaxed);
        }
    }
}

template <typename T, size_t size>
std::optional<T> Queue<T, size>::PopOptional() {
    T element;
    bool result = Pop(element);

    if (result) {
        return element;
    } else {
        return {};
    }
}

其中Pop操作的内存序和push操作的保持一致，这里的问题是pop操作和push操作可能会操作同一个节点的共享数据，在这样的并发场景下，单纯使用const size_t push_count = _data[index].push_count.load(std::memory_order_relaxed);这样宽松的内存序是否满足并发要求。这里使用这样的内存序的原因是在当前的函数逻辑之下，并不需要对_data[index].push_count的push_cout进行修改，即使另一个线程在CPU核心中进行了修改，不进行循环判断也无法实现线程间的同步，所以使用relaxed读取内存中的最新值而不进行同步是合理的。对pop_count的修改需要使用acquire和release的同步语义能获得对节点中push和pop计数的最新值，保证不会对同一个节点进行修改。

std::optional<T> Queue<T, size>::PopOptional()

根据Pop函数的返回值，将结果存储到一个optional中，如果弹出失败则返回值为空。

四、性能分析
上述基于循环数组的无锁数据结构设计上非常巧妙，使用原子变量记录队列中成功pop和push的次数，以此作为队列的队头和队尾的索引，同时为每个节点设置原子的pop和push次数，以此判断该节点中值是否有效，避免了队头和队尾的队列为空和满的判断，同时依靠数组索size_t引作为原子变量，每次加载不需要有依赖管理可以使用更加宽松的内存序，提高程序运行的效率。与基于链表的无锁队列相比，基于数组的实现不需要对内存进行管理，使用栈上的内存由操作系统进行回收，但是同时也会受到队列大小的掣肘，在高并发的实现场景中，选择一个合适的队列大小需要仔细的考虑。
